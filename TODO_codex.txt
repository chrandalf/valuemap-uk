[ ] [L]   Modelled price estimates for sparse 1km cells
          Problem: many 1km cells have only 2-5 sales in any 12-month window, giving
          an unreliable or missing median. These cells show up blank on the map.

          SCIENCE — Ratio Estimator (standard small-area estimation technique):
          For each 1km cell (gx,gy) + (property_type, new_build) combo:
            1. Find the parent 5km cell: parent_gx = gx // 5, parent_gy = gy // 5
            2. Collect N historical year snapshots where BOTH the 1km cell AND
               the parent 5km cell have ≥ 3 sales: pairs (median_1km_t, median_5km_t)
            3. Compute ratio_t = median_1km_t / median_5km_t for each year t
            4. mean_ratio = mean(ratio_t over all valid t)
            5. ratio_cv   = std(ratio_t) / mean_ratio  (coefficient of variation)
            6. estimated_median = mean_ratio × current_5km_median
            Fallback if parent 5km also sparse: climb to 10km parent and repeat.
          Confidence tiers (stored as model_confidence field):
            H  (2) — n_years ≥ 4 AND ratio_cv < 0.15  (stable local premium/discount)
            M  (1) — n_years ≥ 2 AND ratio_cv < 0.30
            L  (0) — fewer than 2 years or high variance (flagged, treat with caution)
          Why this works: it captures the local street-level premium or discount vs
          the wider area. A village that has always been 20% above its 5km average
          will stay flagged at roughly that premium in estimates.
          Why it is not perfect: sudden local changes (new development, HS2, etc.)
          won't be captured until real transactions arrive.

          PIPELINE — new script: pipeline/build_price_model.py
          Input:
          [ ] grid_1km_annual  (already built by make_grid_annual_stack_levels)
          [ ] grid_5km_annual  (already built)
          [ ] grid_10km_annual (already built, used as fallback)
          Steps:
          [ ] Load all three annual stacks from data/model/property/
          [ ] For each (gx_1km, gy_1km, property_type, new_build):
                map to parent 5km cell → join 5km annual stack on same end_month
                collect pairs where both have sales_12m ≥ 3
                compute mean_ratio, n_years, ratio_cv
                if n_years < 2: climb to 10km parent and repeat
                estimated_median = mean_ratio × latest_5km_median
                model_confidence = 2 / 1 / 0 per tiers above
          [ ] Output: data/model/property/grid_1km_modelled_ALL_ALL.json.gz etc.
              Schema per row: {gx, gy, property_type, new_build, end_month,
                               estimated_median, model_confidence, n_years, ratio_cv}
          [ ] Upload: add to upload_model_assets_to_r2.py with --skip-model flag

          API — functions/api/cells.ts
          [ ] New R2 key pattern: grid_1km_modelled_{property_type}_{new_build}.json.gz
          [ ] New in-memory cache: MODEL_CACHE (keyed by grid+type+build)
          [ ] Fetch modelled lookup in parallel inside backfillAll (1km only)
          [ ] New CellRow fields: estimated_median?, model_confidence? (0|1|2)
          [ ] New query param ?modelled=actual|blend|estimated (default: blend)

[ ] [XL]  Current property stock / active listing counts per cell
          - Reddit feedback: "are there 5 houses or 500 houses here?" — users want to
            know how many homes are actively for sale now in a scored area, not just
            how many sold previously. Key for gauging whether a good-scoring area
            actually has anything to buy.
          - What we already have: `tx_count` is historical sales (past 12 months) —
            a rough proxy for turnover but NOT current for-sale inventory
          - The fundamental problem: there is no free, open dataset of current UK
            property listings. Options:
            a) Rightmove / Zoopla / OnTheMarket — all require commercial data
               agreements; not feasible without business arrangement
            b) Scraping portals — ToS violations, unreliable, not viable
            c) DLUHC / Land Registry have no real-time listing data
            d) PropTech APIs (e.g. Homes England, GetAgent, Sprift) — some offer
               listing count data but all paid/commercial
          - Best realistic near-term approximation:
            The EPC dataset (already partially integrated) records every property
            on the market or newly built that had an EPC lodged. It won't give
            "currently listed" but can give total housing stock counts per cell
            (how many known properties of each type exist in the area), which
            answers "is this a big area with lots of houses or a hamlet with 10?"
            This is free open data and we're already loading EPC files in the pipeline.
          - Conclusion: true live listing counts are behind a paywall. Stock count
            from EPC (total properties in area) is achievable and partially answers
            the question. Flag as XL / needs data sourcing investigation.
          - From Reddit: u/Alert_Variation_2579 (2026-03-08)
